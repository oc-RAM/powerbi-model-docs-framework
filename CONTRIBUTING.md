# Contributing

Thank you for considering a contribution to the Power BI Model Docs Framework.

## How to Contribute

### Reporting Issues

- Use GitHub Issues for bugs, template improvements, or missing documentation categories
- Include enough context to reproduce the problem or understand the suggestion
- Check existing issues before opening a new one

### Suggesting Templates or Guides

If you think the framework is missing a template or guide:

1. Open an issue describing the documentation need
2. Explain who would use it and what problem it solves
3. If possible, include a rough outline of what the template should cover

### Submitting Changes

1. Fork the repository
2. Create a branch from `main` with a descriptive name (e.g., `add-rls-documentation-template`)
3. Make your changes
4. Ensure your changes follow the project's writing style and naming conventions
5. Submit a pull request with a clear description of what changed and why

### What Makes a Good Contribution

- **Templates** should be generic enough for broad reuse, not tailored to a specific company or tool
- **Examples** should use the existing Support Operations Analytics model or propose a new generic scenario
- **Guides** should be practical and direct - tell the reader what to do, not what to think about
- **Skills** should solve a real repeated task and follow the existing SKILL.md conventions

## Writing Standards

- Use practical, direct language
- Avoid jargon unless it is standard Power BI / BI terminology
- Keep sentences short
- Use markdown formatting consistently (headers, tables, code blocks)
- Follow kebab-case for file names (e.g., `column-dictionary.md`)
- Follow Title Case for section headers

## Public Safety

All contributions must be public-safe:

- No proprietary content, client names, or company-specific material
- No real table names, measure names, or metric definitions from production systems
- No screenshots or exports from real work environments
- When in doubt, use a generic example

## Review Process

Pull requests are reviewed by project maintainers. We look for:

- Alignment with the framework's scope and design principles (see [PROJECT_SCOPE.md](PROJECT_SCOPE.md))
- Consistency with existing templates and naming conventions
- Public safety compliance
- Clear, practical writing

## Questions

If you are unsure whether a contribution fits, open an issue first to discuss it.
